# Feat 文档写作标准

基于 Diataxis 框架的 Feat 技术文档写作规范。写作或修改 `pages/src/content/docs` 下的文档时遵循本技能。

---

## 核心原则

- **独立思考**：现有文档仅供参考，从读者需求出发重新构思最佳组织方式
- **实用至上**：代码优先，让读者能快速动手实践
- **结构清晰**：参考模板但不被模板束缚，选择最适合当前内容的表现形式
- **真实可靠**：示例优先来自 `demo` 模块的真实可运行代码，也可参考 `feat-test` 模块
- **避免重复**：相同内容只在一个地方详细讲解，其他地方链接引用

## 文档类型速查

| 目的 | 类型 | 人称 |
|------|------|------|
| 新手入门、第一次做某事 | **教程** | 多用「你」 |
| 解决具体问题 | **操作指南** | 主要「你」 |
| 解释设计原理 | **概念解释** | 多用「我们」 |
| API 说明 | **参考文档** | 第三人称 |

## 一、文档体系与内容边界

### 知识分层架构

```
第一层：入门层 - 让新用户在 5 分钟内看到效果
第二层：实践层 - 掌握具体功能的使用方法
第三层：深度层 - 理解底层原理或查阅精确信息
```

### 内容边界规则

| 规则 | 说明 |
|------|------|
| **单一职责** | 每个文档只解决一类问题 |
| **链接引用** | 已在其他文档讲解的内容，使用链接引用而非重复 |
| **渐进深入** | 先出现的文档只讲基础，进阶内容放在后面 |
| **避免重复** | 同样的概念不在多个文档中重复阐述 |

### 交叉引用规范

```md
<!-- ✅ 正确：简要提及 + 链接 -->
本示例使用 `@Controller` 注解，完整用法请参考 [Controller 开发实践](/feat/cloud/controller/)。

<!-- ❌ 错误：在本文档中重复详细讲解 -->
`@Controller` 注解用于标记类为控制器，它有以下参数...
```

## 二、教程写作要点

**适用场景**：面向新手，引导完成完整可运行项目

**常见元素**：引言、前置条件、学习目标、实践步骤、验证结果、总结回顾（根据实际需要选用）

**应该做的**：
- ✅ 聚焦一个具体学习目标
- ✅ 提供可直接复制的完整代码
- ✅ 对涉及的概念只做最简要说明，已有概念提供链接

**不应该做的**：
- ❌ 深入讲解底层原理
- ❌ 罗列所有配置选项
- ❌ 重复前置教程内容

**示例参考**：[cloud/getstart.mdx](pages/src/content/docs/cloud/getstart.mdx)

---

## 三、操作指南写作要点

**适用场景**：解决特定问题，面向已具备基础知识的开发者

**常见元素**：任务描述、前提条件、解决方案、替代方案、注意事项、相关链接（根据实际需要选用）

**应该做的**：
- ✅ 开门见山，直接说明解决什么问题
- ✅ 假设读者已了解基础概念
- ✅ 提供多种方案供选择

**不应该做的**：
- ❌ 从零讲解基础概念
- ❌ 深入解释"为什么"
- ❌ 罗列所有 API 细节

**示例参考**：[cloud/controller.mdx](pages/src/content/docs/cloud/controller.mdx)

---

## 四、概念解释写作要点

**适用场景**：解释框架设计理念、架构决策原因

**常见元素**：背景介绍、设计理念、架构解析、实现细节、方案对比、扩展思考（根据实际需要选用，不必面面俱到）

**应该做的**：
- ✅ 用类比和图示帮助理解
- ✅ 结合代码片段说明原理
- ✅ 分析设计权衡

**不应该做的**：
- ❌ 提供完整可运行的教程式代码
- ❌ 详细罗列 API 参数
- ❌ 重复操作指南中的步骤

**示例参考**：[server/async.mdx](pages/src/content/docs/server/async.mdx)

---

## 五、参考文档写作要点

**适用场景**：API 接口说明、配置项清单

**常见元素**：概述、API 列表、方法详细说明、示例（按信息密度组织，便于查阅）

**方法说明格式**：
```md
### methodName(param1, param2)

方法描述（一句话）

**参数：**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| param1 | String | 是 | 参数说明 |

**返回值：** 返回类型 - 返回值说明

**示例：**
```java
// 简短准确的代码示例
```
```

**应该做的**：信息密度高、表格化呈现、简短准确示例

**不应该做的**：长篇讲解使用场景、解释设计原理、冗长示例代码

## 六、代码示例规范

### 通用要求

1. **来源**：优先使用 `demo` 模块中的真实代码，也可参考 `feat-test` 模块
2. **完整性**：教程和操作指南的代码必须可直接运行（含 main 方法）
3. **注释**：关键逻辑必须有注释

### 代码匹配度标准

| 等级 | 匹配度 | 说明 |
|------|--------|------|
| ✅ 优秀 | 95%+ | 代码完全匹配，可直接复制运行 |
| ⚠️ 良好 | 80%-94% | 核心逻辑正确，少量细节需调整 |
| ❌ 需修正 | <80% | 存在 API 错误或过时代码，必须修复 |

### JDK 8 兼容性（强制）

**允许**：Lambda、Stream API、方法引用、接口默认方法

**禁止**：文本块 `"""`、`var`、Record、增强 switch、JDK 9+ 特性

## 七、格式规范

### 文本格式

| 格式 | 用途 |
|------|------|
| **加粗** | 重要概念、关键词 |
| `代码` | 行内代码、类名、文件名 |
| [链接]() | 内部文档、外部资源 |

### Starlight Aside 组件

```mdx
<Aside type="tip">提示信息</Aside>
<Aside type="caution">注意信息</Aside>
<Aside type="danger">警告信息</Aside>
```

### Mermaid 图表

只在文字难以表达时使用，节点数控制在 10 个以内。

| 内容类型 | 推荐图表 |
|----------|----------|
| 处理流程、时序交互 | `sequenceDiagram` |
| 类继承、组件结构 | `classDiagram` |
| 状态流转 | `stateDiagram` |
| 层级结构、决策树 | `flowchart TD/BT` |

## 八、写作流程

### 写作前
- [ ] 文档类型已确定
- [ ] 目标读者已明确
- [ ] 内容边界已划定
- [ ] 可运行示例已找到

### 完成后
- [ ] 代码示例可运行且符合 JDK 8 规范
- [ ] 标题层级正确（H1→H2→H3）
- [ ] 术语首次出现有解释
- [ ] 人称使用符合文档类型

### 发布前审核
- [ ] **与相关文档交叉检查**：确认无内容重复，如有重复则改为链接引用
- [ ] **链接完整性**：涉及其他文档概念时，已添加正确链接

## 九、内容边界速查

| 文档类型 | 应该详写 | 应该链接 | 应该避免 |
|----------|----------|----------|----------|
| **入门教程** | 环境搭建、第一个程序、验证运行 | 深入功能、概念解释 | API 详细参数、高级配置 |
| **功能教程** | 完成特定功能的完整步骤 | 前置教程、相关概念 | 基础环境配置 |
| **操作指南** | 解决问题的具体步骤 | API 参考、相关教程 | 背景原理、入门知识 |
| **概念解释** | 设计原理、架构分析 | 具体 API 参考 | 操作步骤、入门引导 |
| **参考文档** | 参数列表、方法签名 | 使用教程 | 原理讲解、长篇示例 |

## 十一、常见问题改进

| 问题 | 改进方案 |
|------|----------|
| 先讲原理后给代码 | 教程先展示代码再解释 |
| 示例不完整 | 提供可直接运行的完整代码 |
| 堆砌参数无场景 | 结合具体场景讲解参数用法 |
| 术语无解释 | 首次出现时通俗解释 |
| 内容重复 | 一个地方详细讲解，其他地方链接 |
| 人称混用 | 按类型统一人称 |

---

**核心要诀**：**你的判断比模板更重要。现有文档仅供参考，敢于创造更好的内容。**
