# AI 提示词库

> **核心目标**：为文档编写者提供标准化的 AI 辅助提示词，确保生成的内容符合 Feat 文档规范。

---

## 提示词使用指南

### 基本原则

1. **替换占位符**：将 `{占位符}` 替换为实际内容
2. **提供上下文**：尽可能提供完整的背景信息
3. **迭代优化**：根据输出结果调整提示词
4. **验证输出**：使用 [06-quality-checklist.md](06-quality-checklist.md) 检查生成内容

### 提示词结构

每个提示词包含：
- **用途**：说明提示词的适用场景
- **输入**：需要提供的信息
- **输出格式**：期望的输出结构
- **示例**：实际使用示例

### Prompt-04: 快速入门文档生成

**用途**：生成符合规范的快速入门文档

**提示词模板**：

```

- 适用版本：
- 依赖模块：
- 代码模式：Pattern-XX

## 文档要求

1. 使用快速入门模板
2. 包含 3-5 个步骤
3. 每个步骤包含：操作、代码、验证
4. 包含前置要求检查清单

## 质量检查类提示词

### Prompt-07: 文档质量检查

**用途**：检查文档是否符合 Feat 规范

**提示词模板**：

```
请检查以下文档是否符合 Feat 文档规范：

## 待检查文档

```mdx
{文档内容}
```

## 检查清单

### 代码示例检查
- [ ] 代码来自 demo/ 或 feat-test/，可运行
- [ ] 代码 JDK 8 兼容（无 var、record、文本块等）
- [ ] 包含完整依赖（pom.xml 片段）
- [ ] 每个代码块后有验证步骤
- [ ] 包含常见错误及解决方案

### 文档结构检查
- [ ] 使用标准 Markdown（标题层级 H1→H2→H3，不跳级）
- [ ] 代码块指定语言（java、xml、bash 等）
- [ ] 表格格式正确（表头对齐）
- [ ] 内部链接有效（相对路径正确）

### 内容完整性检查
- [ ] 每个概念包含：定义、使用场景、代码示例
- [ ] API 说明包含参数类型、返回值
- [ ] 版本信息明确（如 Feat ≥ 3.2.0）
- [ ] 术语使用一致

### AI 可解析性检查
- [ ] 使用结构化格式（表格、列表）
- [ ] 代码示例包含文件路径和依赖
- [ ] 反模式包含 ❌/✅ 对比
- [ ] 包含代码模式引用

## 输出格式

### 检查结果摘要
- 总分：X/Y
- 状态：（通过/需修改/不通过）

### 详细检查结果

#### 通过项
1. [检查项] - [说明]

#### 未通过项
1. [检查项] - [问题描述] - [修复建议]

#### 改进建议
1. [建议] - [原因]

### 修复后的文档
[如需要，输出修复后的完整文档]
```

## 术语一致性检查

### Prompt-09: 术语一致性检查

**用途**：检查文档中的术语使用是否一致

**提示词模板**：

```
请检查以下文档中的术语使用是否一致：

## 待检查文档

```mdx
{文档内容}
```

## Feat 标准术语表

| 标准术语 | 别名 | 说明 |
|---------|------|------|
| Router | 路由组件、请求分发器 | |
| Context | 请求上下文 | |
| Interceptor | 拦截器 | |
| Handler | 处理器 | |

## 检查要求

1. 识别所有术语使用
2. 检查是否使用标准术语
3. 检查同一术语是否一致
4. 检查术语首次出现时是否定义

## 输出格式

### 术语使用统计

| 术语 | 使用次数 | 是否标准 | 问题 |
|------|---------|---------|------|
| Router | 5 | ✅ | 无 |
| 路由组件 | 2 | ❌ | 应统一为 Router |

### 不一致问题
1. [位置]：[问题描述] → [建议修改]

### 术语定义建议
建议在文档开头添加：

```mdx
## 术语说明

- **Router**：请求路由分发组件
- **Context**：请求上下文
```
```
