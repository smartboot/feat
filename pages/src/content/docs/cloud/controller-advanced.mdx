前面两章解决了 HTTP 接口的主体：路径、方法、参数和返回值。真实项目还会遇到几类更靠近运行时的问题：当前请求上下文怎么拿，耗时任务怎么返回，大响应怎么压缩，JSON 序列化到底发生在什么时候。

这些能力不应该一开始就全用上。它们更像工具箱：当接口开始碰到状态、异步和性能边界时，再把对应工具拿出来。

## 注入 Session

把 `Session` 声明为方法参数即可注入当前请求的会话对象。这里先看最小用法；如果要判断本地 Session、Redis Session 和认证模型的边界，可以继续看后面的 Session 章节。

```java title="UserController.java"
import tech.smartboot.feat.core.server.Session;

@RequestMapping(value = "/session", method = RequestMethod.GET)
public String session(Session session) {
    String sessionId = session.getSessionId();
    session.put("user", "smartboot");
    String user = session.get("user");
    return "Session ID: " + sessionId + ", User: " + user;
}
```

## 注入 Context

需要直接读取请求头、设置响应头或访问完整 URI 时，注入 `Context`。

```java title="UserController.java"
import tech.smartboot.feat.router.Context;

@RequestMapping(value = "/context", method = RequestMethod.GET)
public String context(Context context) {
    String uri = context.Request.getRequestURI();
    String method = context.Request.getMethod();
    return "URI: " + uri + ", Method: " + method;
}
```

`Context.Request` 和 `Context.Response` 分别是当前请求的 `HttpRequest` 和 `HttpResponse` 对象，字段名首字母大写是框架设计约定。

## 异步响应

当方法内部需要调用远程服务、访问数据库或执行其他耗时操作时，可以返回 `AsyncResponse`，避免阻塞 Feat 的工作线程。

```java title="UserController.java"
import tech.smartboot.feat.cloud.AsyncResponse;
import tech.smartboot.feat.cloud.RestResult;
import java.util.concurrent.CompletableFuture;

@RequestMapping(value = "/async", method = RequestMethod.GET)
public AsyncResponse async() {
    AsyncResponse response = new AsyncResponse();

    CompletableFuture.runAsync(() -> {
        try {
            Thread.sleep(1000);
            response.complete(RestResult.ok("异步响应完成"));
        } catch (InterruptedException e) {
            response.complete(RestResult.fail(e.getMessage()));
        }
    });

    return response;
}
```

<Aside type="tip">
`AsyncResponse.complete(...)` 接收的是 `RestResult`。框架会把这个结果序列化为 JSON 后写回客户端，因此异步接口的响应格式和同步返回 `RestResult` 是一致的。生产环境请把 `CompletableFuture.runAsync` 换成自定义线程池，避免占满 common pool。
</Aside>

## Gzip 压缩

在 `@Controller` 上开启 Gzip，可以压缩返回的大段文本或 JSON，减少网络传输量。默认阈值是 256 字节，低于该值的响应不会压缩，避免小数据量响应因压缩开销而得不偿失。

```java title="ApiController.java"
import tech.smartboot.feat.cloud.annotation.Controller;

@Controller(
    value = "api",
    gzip = true,
    gzipThreshold = 256
)
public class ApiController {

    @RequestMapping(value = "/large", method = RequestMethod.GET)
    public String largeResponse() {
        return "这是一段很长的文本内容，重复多次使其超过 256 字节。".repeat(10);
    }
}
```

Gzip 属性说明：

| 属性 | 类型 | 默认值 | 说明 |
|-----|------|--------|------|
| `gzip` | boolean | false | 是否开启 Gzip 压缩 |
| `gzipThreshold` | int | 256 | 压缩阈值（字节），低于此值不压缩 |

可以用带 `Accept-Encoding: gzip` 的请求验证压缩效果：

```shell title="Gzip 请求"
curl -H "Accept-Encoding: gzip" http://localhost:8080/api/large --compressed
```

## 编译期序列化

Feat Cloud 的 JSON 序列化在编译期生成代码：注解处理器会分析 Controller 方法的返回类型，并为它生成专门的序列化逻辑。

这意味着：

- `String`、`int`、`long`、`boolean` 及其包装类型，以及 `Date`、`Timestamp` 等常用类型，都会生成直接调用的序列化代码。
- 字段访问是直接调用 getter，而不是反射。
- 当对象嵌套超过 4 层、字段类型为 `Object`、或为数组时，会回退到 FastJSON2 进行序列化。

以下类型会享受编译期优化：

| 类型 | 序列化方式 |
|------|-----------|
| `String`、基本类型、包装类型 | 直接写输出流 |
| `Date`、`Timestamp` | 直接格式化后写输出流 |
| 自定义 POJO | 按字段生成 getter 调用 |
| `List`、`Map` | 生成遍历代码 |
| 数组、`Object`、嵌套超过 4 层 | 回退 FastJSON2 |

<Aside type="caution">
编译期序列化只覆盖同步返回的对象。`AsyncResponse` 返回的 `RestResult` 是在运行时用 FastJSON2 序列化的，因为它在编译期无法确定异步完成时的具体结果。
</Aside>

## 如何判断该用哪一个

如果只是读写一个普通资源，前两章的 `@RequestMapping`、`@Param`、`@PathParam` 已经足够。  
只有出现下面这些信号时，再引入本章能力：

- 需要读取完整请求或直接设置响应头时，用 `Context`
- 需要在多次请求间保存少量状态时，用 `Session`
- 当前请求会等待远程调用或较长任务时，用 `AsyncResponse`
- 返回体经常超过几百字节且客户端支持压缩时，再考虑 Gzip
