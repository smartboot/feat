# Feat 文档写作标准

为了确保 Feat 文档的一致性、专业性和易读性，特制定以下写作标准。所有参与 Feat 文档编写的 AI 应遵循这些准则，以产出符合项目要求的高质量文档。

## 一、文档结构标准

### 1.1 文档头部信息
每篇文档必须包含标准的头部信息（Frontmatter）：

```md
---
title: 文档标题
description: 简短描述文档内容
sidebar:
  order: 排序数字（可选）
---
```

### 1.2 标题层级
- 以存在文章title,无需使用 `#` 表示主标题（H1）
- 使用 `##` 表示章节标题（H2）
- 使用 `###` 表示小节标题（H3）
- 最多使用三级标题，避免更深的嵌套

### 1.3 内容组织原则
1. **概述先行**：开头应有简要介绍，说明本文档的目的和主要内容
2. **循序渐进**：从基础概念开始，逐步深入
3. **实例辅助**：重要概念和功能应配有代码示例或图示
4. **总结回顾**：结尾可提供要点总结或下一步指引

## 二、语言风格标准

### 2.1 语气与语调
- **专业而不晦涩**：使用准确的技术术语，但避免过度复杂的表达
- **客观且友好**：保持中立态度，同时体现人文关怀
- **简洁明了**：用最少的文字传达最多的信息

### 2.2 表达规范
- 使用主动语态，避免被动语态
- 使用现在时态描述一般情况
- 保持句子结构简单，避免过长的复合句

### 2.3 词汇选择
- 使用统一的专业术语（如："Feat"而非"feat"）
- 避免口语化表达
- 优先使用具体而非抽象的词汇

## 三、复杂内容处理标准

面对复杂的技术概念或实现细节，文档需要更加详细的解释，同时保持清晰的段落结构：

### 3.1 分解复杂概念
- 将复杂概念分解为多个简单的子概念
- 每个子概念独立成段，便于理解
- 使用小标题标识每个子概念或步骤

### 3.2 层次化解释
- 先提供高层次的概览，再逐步深入细节
- 使用"总-分"结构组织内容
- 在进入下一层次前确保当前层次已被充分解释

### 3.3 多媒体辅助说明
- 对于复杂流程，使用流程图或架构图进行可视化展示
- 提供实际代码示例来演示关键点
- 必要时使用表格对比不同方案或参数

### 3.4 实际案例分析
- 通过具体案例来解释抽象概念
- 提供常见问题及解决方案
- 展示最佳实践和反面模式

### 3.5 渐进式学习路径
- 标注内容的难度级别（基础、进阶、专家）
- 提供前置知识要求说明
- 设置练习或思考题帮助巩固理解

## 四、技术内容标准

### 4.1 代码示例规范
- 每段代码示例必须配有简要说明
- 重要的代码行应添加注释
- 示例代码应具备完整性和可执行性
- **突出重点**：仅展示与当前讨论主题相关的核心代码，避免冗长的样板代码
- 使用适当的语法高亮标记

```java
// 精简的核心示例：突出关键逻辑
public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello, Feat!"); // 关键功能点
    }
}
// 注：省略了不影响理解的样板代码
```

所有代码示例都应该链接到仓库中的实际文件，以便读者可以查看完整的上下文和运行示例。例如：

有关完整示例，请参见 [HelloWorld.java](https://gitee.com/smartboot/feat/blob/master/feat-test/src/main/java/tech/smartboot/feat/demo/HelloWorld.java)

:::note
如果文档中的代码示例需要提供完整可运行的代码，应当在文档中添加指向仓库中实际文件的链接，方便读者查找和运行完整示例。
代码统一存放在：https://gitee.com/smartboot/feat/blob/master/feat-test/src/main/java/tech/smartboot/feat/docs/ 目录下。
:::

### 4.2 图表使用规范
- **首选 Mermaid**：对于流程图、架构图等，优先使用 Mermaid 语法
- **SVG 备选方案**：当 Mermaid 无法满足需求时，可使用 SVG 格式并保存到 img 目录
- 图表应有明确的标题和说明文字
- 图片文件应存储在相应的 img 目录中
- 使用相对路径引用图片
- 使用 Mermaid 时，必须在文档顶部导入自定义组件：`import Mermaid from '../../../components/Mermaid.astro';`

#### Mermaid 图表示例：
使用 Mermaid 组件展示图表：

在使用 Mermaid 图表之前，需要在文档顶部导入自定义的 Mermaid 组件：

```md
import Mermaid from '../../../components/Mermaid.astro';
```

然后在文档正文中使用 Mermaid 组件：

```md
{/* Mermaid 流程图示例 */}
<Mermaid code={`
graph TD
    A[开始] --> B[处理]
    B --> C[结束]
`}/>
```

#### SVG 图表示例：
对于复杂或特殊要求的图表，使用 SVG 格式：

```md
{/* SVG 图表示例 */}
<img src="./img/example.svg" alt="示例图表" width="100%"/>
```

### 4.3 链接引用规范
- 内部链接使用相对路径
- 外部链接应在新窗口打开
- 链接文本应具有描述性，避免使用"点击这里"

## 五、格式排版标准

### 5.1 强调样式
- **加粗**：用于强调关键词或重要概念
- *斜体*：用于首次提及的术语或书籍名称
- `代码`：用于行内代码或命令

### 5.2 列表使用
- 有序列表用于步骤说明
- 无序列表用于列举要点
- 列表项应保持平行结构

### 5.3 表格规范
- 表头使用简洁明确的标签
- 表格内容应左对齐
- 表格前后应有说明文字

## 六、特殊内容处理

### 6.1 注意事项与警告
对于需要特别提醒的内容，使用专门的提示框：

:::note
这是一般性提示信息
:::

:::caution
这是需要注意的警告信息
:::

### 6.2 版本兼容性说明
涉及版本差异的内容应明确标注适用版本范围

### 6.3 性能与安全建议
提供性能优化建议和安全注意事项

## 七、文档质量检查清单

在完成文档编写后，请对照以下清单进行检查：

- [ ] 标题是否准确反映内容主旨
- [ ] 是否包含必要的前置知识说明
- [ ] 代码示例是否完整且可运行
- [ ] 是否存在错别字或语法错误
- [ ] 链接是否有效
- [ ] 图片是否正常显示
- [ ] 内容是否有逻辑错误或表述不清之处
- [ ] 是否符合整体文档风格

## 八、AI 写作特别指导

### 8.1 内容原创性
- 基于 Feat 项目实际情况进行描述
- 避免直接复制现有文档内容
- 确保技术细节的准确性

### 8.2 上下文理解
- 准确理解用户查询意图
- 结合 Feat 框架特点进行阐述
- 考虑目标读者的技术水平

### 8.3 个性化表达
- 在遵循规范的前提下，适度体现个性
- 适当使用比喻帮助理解复杂概念
- 保持积极正面的态度

通过遵循以上标准，AI 编写的 Feat 文档将更加专业、一致且易于理解，有助于提升整个项目的文档质量和用户体验。
